加载性能优化 · 技术分析

ES 模块瀑布问题诊断 + 两步修复方案
影响范围：index.html · style.css · main.js 改动量：极小（约 25 行） 不涉及游戏逻辑

一、问题诊断

项目总文件量约 100 KB（JS + CSS 合计），并不大。加载变慢的原因不是带宽，而是 ES 模块的串行发现机制。

浏览器加载 ES 模块的规则：必须先下载并解析一个文件，才能发现它的 import，才能发起下一层请求。现有项目形成了 3 级瀑布：

Level 0　index.html → main.js ← 1 次往返（RTT）
Level 1　解析 main.js，发现 11 个 import，并行请求 ← 第 2 次 RTT
cards.js · species.js · config.js · gameState.js · storage.js
photoSequence.js · fieldGuide.js · rarityDisplay.js · spotManager.js
focusEngine.js → focusConfig.js
gameSession.js → birdManager.js · encounterSystem.js · cardDraw.js · spots.js
Level 2　解析 gameSession / focusEngine，发现第 2 批 import ← 第 3 次 RTT

总请求：17 个 JS 文件　　级联层数：3 级　　实测延迟估算：3 × RTT ≈ 150–600 ms（RTT = 50–200ms）
慢的不是文件大小，是"发现 → 请求 → 发现 → 请求"的串行等待。每增加一个 import 层级就多付出一次往返时延。随着项目继续增大（新鸟种数据、更多模块），这个问题只会加重。

二、修复方案（两步，可独立实施）

1
modulepreload — 消除级联
实质性性能提升
在 index.html 的 <head> 中为全部 17 个 JS 文件添加 <link rel="modulepreload">。

效果：浏览器解析 HTML 时就立刻并行预取所有模块，无需等待 main.js 执行后再逐层发现。三级串行 → 一次并发批量请求，理论上节省 2 次 RTT。

兼容性：所有现代浏览器（Chrome 66+、Safari 17+、Firefox 115+）均已支持。不支持的浏览器直接忽略该标签，不影响功能。
2
加载遮罩 — 覆盖空白期
体验改善
在 HTML 里放一个初始可见的全屏遮罩，用纯 CSS 展示（无需 JS），JS 初始化完成后再淡出移除。

效果：用户打开链接后立刻看到内容（加载文字），而不是白屏。即使已经加了 modulepreload，CSS 解析和首次渲染之间也有短暂空白，遮罩补上这段。

本质：不加快加载，只是把"白屏"换成"有内容的屏"，体感更流畅。
两步可独立实施，但建议一起做。modulepreload 解决实际速度，加载遮罩解决视觉体验——后者在项目继续增大时也是长期需要的保险。

三、实现规格

3.1　index.html — 添加 modulepreload + 遮罩 HTML

<head> 末尾（CSS link 之后）加入全部 17 个模块的预加载声明：

<!-- modulepreload：并行预取全部模块，消除级联等待 --> <link rel="modulepreload" href="./src/main.js"> <link rel="modulepreload" href="./src/gameSession.js"> <link rel="modulepreload" href="./src/gameState.js"> <link rel="modulepreload" href="./src/birdManager.js"> <link rel="modulepreload" href="./src/encounterSystem.js"> <link rel="modulepreload" href="./src/photoSequence.js"> <link rel="modulepreload" href="./src/focusEngine.js"> <link rel="modulepreload" href="./src/fieldGuide.js"> <link rel="modulepreload" href="./src/cardDraw.js"> <link rel="modulepreload" href="./src/rarityDisplay.js"> <link rel="modulepreload" href="./src/spotManager.js"> <link rel="modulepreload" href="./src/storage.js"> <link rel="modulepreload" href="./data/cards.js"> <link rel="modulepreload" href="./data/species.js"> <link rel="modulepreload" href="./data/config.js"> <link rel="modulepreload" href="./data/spots.js"> <link rel="modulepreload" href="./data/focusConfig.js">

<body> 内 <main> 之前插入遮罩 div（CSS 默认可见，不依赖 JS 展示）：

<div id="app-loading"> <p class="app-loading-text">正在加载…</p> </div>
维护注意：后续新增 JS 文件时，同步在 index.html 添加对应的 modulepreload 标签，否则新文件仍会回退到串行加载。

3.2　style.css — 遮罩样式（追加到文件末尾）

/* ── 加载遮罩 ── */ #app-loading { position: fixed; inset: 0; background: #f4f0e8; /* 与 body 背景一致 */ display: flex; align-items: center; justify-content: center; z-index: 9999; transition: opacity 0.25s ease; } #app-loading.is-hiding { opacity: 0; pointer-events: none; } .app-loading-text { font-family: "Noto Sans SC", "Microsoft YaHei", Arial, sans-serif; font-size: 13px; color: #8a7f6a; letter-spacing: 0.12em; }

3.3　src/main.js — 初始化完成后淡出遮罩（文件末尾）

文件末尾现有 render() 调用后，追加以下代码。requestAnimationFrame 确保先等游戏界面完成一帧渲染，再开始遮罩淡出。

render(); // 初始化完成后淡出加载遮罩 const _loadingEl = document.getElementById('app-loading'); if (_loadingEl) { _loadingEl.addEventListener('transitionend', () => _loadingEl.remove(), { once: true }); requestAnimationFrame(() => _loadingEl.classList.add('is-hiding')); }
