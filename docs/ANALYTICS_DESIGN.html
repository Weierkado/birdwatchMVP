一、整体架构

游戏客户端（浏览器） 游戏运行期间，将每个事件对象 push 到内存中的 sessionEvents[] 数组。不实时上报，避免网络请求干扰游戏体验。
局末批量 POST 游戏结束进入结算（SETTLEMENT）时，将本局所有事件 + 问卷答案打包成一个 JSON，POST 到 Apps Script Web App URL。
Apps Script 接收 & 写入 Apps Script 的 doPost() 解析 JSON，将事件流写入 raw_events 表，将问卷答案写入 survey_answers 表。
Looker Studio 读取展示 Looker Studio 连接 Google Sheets，看板实时刷新。详见 DASHBOARD_DESIGN.html。
批量上报而非实时上报的原因：避免每次按快门都发请求，影响手感；万一网络不稳也不会中断游戏。代价是如果玩家直接关掉浏览器而没有走结算流程，该局数据丢失——可以接受，因为问卷数据同样会丢失，这局数据的完整性已经不高。
组件选型说明
事件存储（客户端）JS 数组（内存）挂在 gameState 或独立的 analytics 模块上，局末随 state 一起处理
上报接口Apps Script Web App免费，部署后得到一个 HTTPS URL，直接 fetch POST 即可
数据持久化Google Sheets两张表：raw_events / survey_answers
可视化Looker Studio连接同一个 Sheets 文件

二、测试入口与数据收集流程

测试版新增专用入口，区别于普通开始游戏，玩家自主选择进入。

START 界面 新增测试专用按钮（如"参与测试 · 留下反馈"），与正常"开始游戏"并列。点击测试按钮才进入埋点流程；直接开始游戏不触发 ID 输入和 Q0，但其他埋点事件仍然记录（tester_id 留空）。
ID 输入面板（新模式：TESTER_ID_INPUT） 玩家输入测试者昵称/ID。记录到 analytics.testerId，后续所有事件携带此值。
身份选择面板（新模式：TESTER_PROFILE） Q0 单选题（4 个观鸟阅历选项）。选择后记录到 analytics.testerLevel（1–4），进入 START_SPOT_SELECT 正常开始游戏。
游戏进行中 每次关键操作触发对应事件，push 到 analytics.events[]。
SETTLEMENT 结算界面 进入结算时，先触发 session_end 事件入队，然后显示游后问卷（Q1–Q12，可整体跳过）。
批量上报（问卷提交或跳过后） 玩家提交问卷或点击跳过后，将 analytics.events[] + analytics.survey 打包 POST。上报完成后清空本局缓存，回到 START 界面。
问卷跳过时：问卷字段留空或标记 surveySkipped: true，事件数据仍然正常上报。不要因为玩家跳过问卷就丢弃行为数据。

三、P0 事件（必须实现）

这六个事件是数据分析的最低完整性保证，测试包上线前必须全部就位。

session_start
P0 必须
触发时机：玩家选定初始鸟点、游戏正式进入 EXPLORE 模式的瞬间（即 startGameAtSpot() 执行后）。
字段类型说明
event_typestring"session_start"（固定值）
timestampstringISO 8601，new Date().toISOString()
session_idstring本局唯一标识，建议用 Date.now() + Math.random() 生成，整局所有事件携带同一个值
tester_idstring玩家填写的 ID；从测试入口进入时有值，直接开始游戏时为空字符串
tester_levelnumber 1–4Q0 选项序号（1=完全不了解 … 4=资深观鸟者）；跳过测试入口时为 0
tester_level_textstringQ0 选项原文，方便 Sheets 里直接读
start_spot_idstring初始鸟点 ID（如 "pond_edge"）
max_turnsnumber本局回合上限，当前固定 30，记录方便后续对比不同配置
max_photosnumber本局 SD 卡上限，当前固定 15
session_end
P0 必须
触发时机：游戏进入 SETTLEMENT 模式时（包含回合用尽、SD 卡满、主动撤退三种路径），在显示结算 UI 之前入队。
字段类型说明
event_typestring"session_end"
timestampstringISO 8601
reasonstring"turns_exhausted" / "sd_full" / "retreat"
final_turnnumber结束时的当前回合数
total_photosnumber本局共拍照片数
unique_speciesnumber本局拍到的不同鸟种数
spots_visitednumber本局去过的不同鸟点数
duration_msnumbersession_end.timestamp − session_start.timestamp，毫秒
photo_enter
P0 必须
触发时机：enterPhotoMode() 执行时（玩家发现鸟并进入 PHOTO 模式）。
字段类型说明
event_typestring"photo_enter"
timestampstringISO 8601
species_idstring目标鸟种 ID
spot_idstring当前鸟点 ID
current_turnnumber当前回合数
sd_remainnumber当前 SD 卡剩余格数（进入拍照前）
photo_shoot
P0 必须
触发时机：玩家按快门且成功拍到照片时（在 handlePhotoAction 的 shoot 分支中，card 有效且未满 SD 卡时）。
字段类型说明
event_typestring"photo_shoot"
timestampstringISO 8601
species_idstring目标鸟种 ID
behavior_statestring"NORMAL" / "INTERESTING" / "REMARKABLE"
card_raritystring抽到卡牌的稀有度，来自 card.rarity
is_new_cardboolean是否是首次获得该卡牌
spot_idstring当前鸟点 ID
current_turnnumber当前回合数
sd_remainnumber拍照后剩余 SD 卡格数
shoot_indexnumber本次拍照邂逅中第几次按下快门（从 1 开始）
photo_wait
P0 必须
触发时机：玩家在 PHOTO 模式选择"等待"时（handlePhotoAction 的 wait 分支），advancePhotoSequence 执行后。
字段类型说明
event_typestring"photo_wait"
timestampstringISO 8601
species_idstring目标鸟种 ID
from_statestring等待前的行为时段
to_statestring等待后的行为时段（含 "FLY_AWAY"）
current_turnnumber当前回合数
photo_end
P0 必须
触发时机：离开 PHOTO 模式时（exitPhotoMode 或 enterSettlementFromPhotoMode 执行前），涵盖飞走、主动放弃、SD 卡满三种出口。
字段类型说明
event_typestring"photo_end"
timestampstringISO 8601
species_idstring目标鸟种 ID
exit_reasonstring"fly_away" / "give_up" / "sd_full"
photos_in_encounternumber本次邂逅拍摄的照片数（0 = 一张没拍就走了）

五、问卷数据结构

问卷答案独立于事件流，作为 analytics.survey 对象存储，随批量上报一起发送，写入 survey_answers 表。

// analytics.survey 结构 { "q0": 2, // 游前身份选择，1–4；跳过测试入口时为 0 "q0_text": "了解观鸟，但还没开始实践", "q1": 2, // Q1–Q8：选项序号（1=第一项，依此类推）；跳过时为 null "q2": 1, "q3": 3, "q4": 2, "q5": 2, "q6": 1, "q7": 4, "q8": 1, "q9": 8, // NPS 分，0–10 "q10": "我觉得等鸟出现精彩动作的时候按", // 描述题，原文字符串 "q11": "鸺鹠", // 开放题 "q12": "希望有更多鸟点", "survey_completed": true, // 玩家是否提交了问卷（false = 整体跳过） "survey_skipped": false }
Q0 在何时写入 survey：玩家完成身份选择（TESTER_PROFILE 模式）时立即写入 analytics.survey.q0 和 analytics.survey.q0_text。Q1–Q12 在局后问卷界面交互时逐一写入。上报时所有字段一起发送，无需等到问卷全部完成。

六、批量上报数据包格式

每局游戏结束后，POST 一个 JSON 对象到 Apps Script URL。

// POST body，Content-Type: application/json { "session_id": "1747123456789_0.382", "tester_id": "DEV_YX", "tester_level": 2, "events": [ { "event_type": "session_start", "timestamp": "2026-05-15T10:23:01.456Z", "session_id": "1747123456789_0.382", "tester_id": "DEV_YX", "tester_level": 2, "start_spot_id": "pond_edge", "max_turns": 30, "max_photos": 15 }, // ... 更多事件 ... { "event_type": "session_end", "timestamp": "2026-05-15T10:35:22.001Z", "session_id": "1747123456789_0.382", "reason": "turns_exhausted", "final_turn": 30, "total_photos": 9, "unique_species": 3, "spots_visited": 2, "duration_ms": 740545 } ], "survey": { "q0": 2, "q0_text": "了解观鸟，但还没开始实践", "q1": 1, "q2": 1, "q3": 2, "q4": 2, "q5": 2, "q6": 1, "q7": 4, "q8": 1, "q9": 8, "q10": "等鸟出现好时机", "q11": "鸺鹠", "q12": "更多鸟点", "survey_completed": true, "survey_skipped": false } }
每个事件都携带 session_id 和 tester_id 作为冗余字段——这样 Apps Script 展开成行后，每行都是自描述的，不需要 JOIN 才能知道这个事件属于谁。

七、Apps Script 端点实现

在 Google Sheets 绑定的 Apps Script 编辑器里创建一个 Web App，部署后获得 URL。

// Code.gs（Apps Script） function doPost(e) { try { const payload = JSON.parse(e.postData.contents); const ss = SpreadsheetApp.getActiveSpreadsheet(); // 1. 写入事件流到 raw_events 表 const eventsSheet = ss.getSheetByName("raw_events"); payload.events.forEach(event => { eventsSheet.appendRow([ event.timestamp, payload.session_id, payload.tester_id, payload.tester_level, event.event_type, event.species_id || "", event.spot_id || "", event.reason || "", event.behavior_state || "", event.found !== undefined ? String(event.found) : "", event.sd_remain !== undefined ? event.sd_remain : "", event.current_turn || "", event.duration_ms || "", event.exit_reason || "", event.photos_in_encounter !== undefined ? event.photos_in_encounter : "", event.shoot_index || "", event.is_new_card !== undefined ? String(event.is_new_card) : "", event.card_rarity || "", event.travel_cost || "", event.from_spot_id || "", event.to_spot_id || "", JSON.stringify(event) // 最后一列保存完整原始 JSON，方便补救 ]); }); // 2. 写入问卷答案到 survey_answers 表 const surveySheet = ss.getSheetByName("survey_answers"); const sv = payload.survey || {}; surveySheet.appendRow([ new Date().toISOString(), payload.session_id, payload.tester_id, payload.tester_level, sv.q0 || "", sv.q0_text || "", sv.q1 || "", sv.q2 || "", sv.q3 || "", sv.q4 || "", sv.q5 || "", sv.q6 || "", sv.q7 || "", sv.q8 || "", sv.q9 !== undefined ? sv.q9 : "", sv.q10 || "", sv.q11 || "", sv.q12 || "", sv.survey_completed ? "TRUE" : "FALSE", sv.survey_skipped ? "TRUE" : "FALSE" ]); return ContentService .createTextOutput(JSON.stringify({ status: "ok" })) .setMimeType(ContentService.MimeType.JSON); } catch (err) { return ContentService .createTextOutput(JSON.stringify({ status: "error", message: err.message })) .setMimeType(ContentService.MimeType.JSON); } }
部署设置：Apps Script → 部署 → 新建部署 → 类型选"Web 应用" → 执行身份选"我" → 访问权限选"所有人（包括匿名）"。部署后复制 URL，写入游戏配置文件的 ANALYTICS_ENDPOINT 常量。

蓝色列为关联键，绿色列为跨表公用的身份字段。Looker Studio 通过 session_id 做表间 JOIN，通过 tester_id 追踪同一玩家的多局行为。

九、开发者本地测试数据过滤

开发者自测产生的数据需要与真实玩家数据隔离，约定 tester_id 前缀即可，无需在游戏里加额外开关。

使用场景建议 tester_id
日常功能验证DEV_YX（固定用这一个）
数值压测 / 多局连打DEV_stress
Bug 复现DEV_debug_1 等临时 ID

在 Looker Studio 里，为所有图表加一条后台静默过滤器：tester_id 不包含 "DEV_"。审查自测记录时临时改为 包含 "DEV_" 即可切换视角。

十、本地开发 / 无网络 Fallback

上报接口 URL 放在配置文件里，未配置时静默降级，不影响正常游戏流程。

// data/config.js 里新增 export const ANALYTICS_ENDPOINT = ""; // 留空 = 仅本地 console.log；填入 Apps Script URL 后上报 // analytics.js 里的上报函数 async function submitSession(payload) { if (!ANALYTICS_ENDPOINT) { console.log("[Analytics] 本地模式，数据包如下：", payload); return; } try { await fetch(ANALYTICS_ENDPOINT, { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify(payload) }); } catch (err) { console.warn("[Analytics] 上报失败：", err.message); // 不抛出，游戏继续正常运行 } }
本地开发时 ANALYTICS_ENDPOINT 留空，控制台打印完整数据包，可以直接验证事件结构是否正确。测试包发布前，在配置文件里填入 Apps Script URL 即完成上报接入，无需改动任何业务逻辑。
