一、为什么要做双轨

5.28 拍照-发送-等回复 循环里，原 30s 固定延迟有三个问题：

  • 路人节奏拖：对观鸟无感的玩家拍完照看不到反馈，30s 焦虑窗口太长，可能放下手机
  • chatbot 化诱惑：但如果一刀切到 2-10s，妹妹回得像 always-on chatbot，破坏 SistersChat_Skill 第 24 行明确写的"她要上课、补课、写作业，不能随时看手机"的人设核心
  • 2 条文案前提失效：after_class_001 和 soft_001 的"压在书底下/还在写题"在 5s 后弹出会很违和

双轨的解法：

QUICK 妹妹现在有空
2-10s 随机延迟 · 占比 ~50%
日常轻快回复。"刚看到""能认能认""噗有点糊"这种没有明确情境前提的台词，统统走这条。
体感近乎即时，玩家不会忘
承担循环爽快感、风味展示
DELAYED 妹妹刚才在做别的事
30-60s 随机延迟 · 占比 ~50%
延迟解释型回复。"刚补习完""被妈叫去拿快递""体育课刚结束"——延迟变成内容本身。
体感等了一会儿但等出了"哦原来她在干这个"的钩子
承担世界观窗口、人设质感锚点
把"延迟"从节奏参数升级为叙事工具。30s 固定延迟里 30s 是惩罚；双轨里 30s 是奖励——它换来的是妹妹另一个真实生活的浮现。

二、设计原则

  1. 延迟必须有理由。delayed 池每条文案都要明确说出"刚才在做什么"，让玩家"哦"一声。没理由的慢回是惩罚，有理由的慢回是质感
  2. 双池独立存活。quick 和 delayed 各自必须能在所有 condition 组合下找到候选，互不依赖
  3. 初二真实素材。情境必须是初二学生的日常（学校 / 补习 / 家 / 食堂 / 体育课），不写成都市白领或大学生
  4. 不要每条都加 emoticon。delayed 文案里的"忙"和"慌"自带情绪，emoticon 反而稀释——保留改写后的整体节奏：~ 50% 条目带 emoticon
  5. 偶尔触发上学/睡前等强时段标记。这些受 timeOfDay 限定，但占少量。大部分 delayed 文案 timeOfDay 通用

三、数据结构变更

现有每条消息的 delay 字段：

"delay": { "mode": "fixed", "seconds": 30 }

新结构：在消息根级加 delayProfile 字段，delay 字段保留但不再是主控制源。

// 一条 quick 消息（默认） { "id": "liya_photo_reply_blurry_001", "delayProfile": "quick", "delay": { "mode": "profile" }, // ... 其余字段不变 ... } // 一条 delayed 消息 { "id": "liya_photo_reply_after_class_001", "delayProfile": "delayed", "delay": { "mode": "profile" }, // ... 其余字段不变 ... }

实际秒数由 delayProfile + 全局 config 决定，不再写死在每条消息里。

向后兼容：没有 delayProfile 字段的旧消息默认按 quick 处理。delay.mode 还可以保留 "fixed" 作为 override（极个别消息可写死特定秒数），但本版不使用。

四、选择算法（profile-first）

function selectLiyaPhotoReply(eventName, context, options) { // 1. Roll profile based on stable seed const profile = rollProfile(eventName, context, options); // profile ∈ {"quick", "delayed"} // 2. Filter by profile + condition + stage（沿用现有 matchesConditions） const pool = messages.filter(m => m.delayProfile === profile && m.trigger.event === eventName && matchesConditions(m.conditions, context) && (!m.stage || m.stage === stage) ); // 3. Fallback: 如果该 profile 池子空，用另一个 profile const finalPool = pool.length > 0 ? pool : messages.filter(...alt profile); // 4. 在最终池里按 priority + specificity 选（沿用现有逻辑） return selectByPrioritySpecificity(finalPool, context, options); } function rollProfile(eventName, context, options) { const seed = createLiyaSelectionSeed(eventName, context, options) + "|profile"; const hash = hashStringToUint32(seed); const ratio = config.MESSAGE_DELAY_PROFILE_RATIO ?? 0.5; return (hash / 0xFFFFFFFF) < ratio ? "delayed" : "quick"; } function resolveDelaySeconds(message) { if (message.delay?.mode === "fixed") return message.delay.seconds; const [min, max] = message.delayProfile === "delayed" ? [config.MESSAGE_DELAY_LONG_MIN, config.MESSAGE_DELAY_LONG_MAX] : [config.MESSAGE_DELAY_QUICK_MIN, config.MESSAGE_DELAY_QUICK_MAX]; return randomInt(min, max); }
关键决策选择原因
Profile roll 时机选消息之前先 roll profile 再选——保证 50/50 分布稳定，不被消息池大小扭曲
Roll 用什么 seed沿用现有 selection seed + "|profile"同一张照片的 profile 决策可重现，方便 debug
Profile 池空怎么办fallback 到另一个 profile极端情况（比如 condition 太窄两边都没匹配）才会触发；记录 warning 日志
实际秒数min/max 之间均匀随机不用高斯，避免大多数集中在中间值

五、配置项与边界

新增到 data/config.js：

// 妹妹回复延迟双轨配置 export const MESSAGE_DELAY_PROFILE_RATIO = 0.5; // delayed 占比，默认五五开 export const MESSAGE_DELAY_QUICK_MIN = 2; // quick 池最小延迟（秒） export const MESSAGE_DELAY_QUICK_MAX = 10; // quick 池最大延迟（秒） export const MESSAGE_DELAY_LONG_MIN = 30; // delayed 池最小延迟（秒） export const MESSAGE_DELAY_LONG_MAX = 60; // delayed 池最大延迟（秒）
边界处理
玩家短时间连发多张照片每张照片独立 roll profile，独立计算延迟。可能出现 3 张连续 delayed 的情况——这是允许的（妹妹真的"那段时间一直忙"）
delayed 消息抵达时玩家已经退出本局下一局打开聊天界面时该条消息正常显示。session_end 时记 session_end.pending_delayed_count
同一张照片选到的消息 delayProfile 与 roll 出来的 profile 不匹配（fallback 触发）实际秒数按消息自己的 delayProfile 算（不是按 roll 的 profile），保证文案和延迟语义一致
玩家在等待期再拍一张不影响正在排队的消息——每张照片独立的延迟计时器

第七节会再加 18 条 delayed 文案，使总比例达到 quick 37 / delayed 20，与 50/50 ratio 配合时实际选中比例约 35:65 偏 delayed——但 ratio 控制的是每张照片的 profile 决策，不是池大小，所以最终触发比例严格按 0.5。

如果发现某种 condition（如 firstTimeSpecies + clear + centered）在 delayed 池没匹配项，会 fallback 到 quick——延迟也按 quick 算。这是允许的：第一次拍到清晰居中的照片本来就该立刻被夸，没必要刻意延迟。

八、与埋点的交互

ANALYTICS_DESIGN_V2 已经在 sister_message_received 事件里收集了 delay_seconds。本设计不需要新增字段，但需要让 BI 层能区分两个池：

派生指标计算用途
quick / delayed 实际触发比按 delay_seconds 二分（<15s 算 quick，否则 delayed），算各自占比验证 ratio 0.5 是否实际生效，校准随机种子分布
quick 池阅读率sister_replies_read / total 在 delay_seconds < 15s 子集验证短延迟是否带来更高阅读率
delayed 池阅读率同上但在 delay_seconds ≥ 30s 子集验证"等了一分钟玩家有没有去看"——如果显著低于 quick 池，说明 30-60s 还是太长
delayed 池印象最深消息Q8 印象最深消息 message_id 与 delayed 池交集看玩家在问卷里印象最深的是不是 delayed 文案——是的话证明这条策略有效
本版埋点已经能测这一切，不需要额外开发。看板第 4 页（H3 风味）加 1 个对比图即可。

九、测试期观察点

  • 红线沿用 ANALYTICS_DESIGN_V2 红线 1：阅读率 < 50% 仍是失败。但本版要分别看 quick / delayed 两子集
  • delayed 池阅读率显著低（比 quick 池低 15pp+）：说明 30-60s 确实太长，下版考虑调到 20-40s
  • Q5 选项分布：选"开小差刷其他东西"的占比在 quick / delayed 池比例发生变化时是否同步变化
  • 访谈关键问题："妹妹有时候过一阵才回，你怎么看？"——观察玩家是否能识别"她在另一个生活"还是觉得"游戏卡了/不稳定"
  • Q8 印象最深：如果 top 5 里 ≥ 2 条来自 delayed 池，说明"延迟即叙事"成立。如果 0 条，说明 delayed 文案设计还需打磨

十、message-editor 兼容性

编辑器需要小改动：

  • 消息编辑面板加一个 delayProfile 下拉选择（quick / delayed），默认 quick
  • 触发情况总览的 smoke test context 增加一个 profile 维度——同一 condition 矩阵在 quick / delayed 下分别走一遍
  • 校验：每条消息必须有 delayProfile 字段；缺失时报 warning（不阻断保存）
  • 导入旧 JSON（无 delayProfile）时自动按 quick 标注，给 dev 一个提醒
编辑器作业流程不变：导入 → 编辑 → 导出 → 替换 data/liyaMessages.json → 进游戏验证。本变更对作业流是透明扩展。
