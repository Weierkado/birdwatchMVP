力娅消息编辑器使用文档

用于维护 data/liyaMessages.json：导入、编辑、校验、查看触发情况、导出，再手动替换正式数据文件。

内存草稿 不自动保存 不写游戏存档 需要本地服务器打开

1. 这个编辑器是做什么的

力娅消息编辑器是一个开发用工具页，用来编辑和检查 data/liyaMessages.json 中的力娅消息文本。它不会自动修改项目文件，也不会影响正式游戏存档。

一句话：它是“文本策划表 + 触发检查器”，不是正式游戏界面。
能力说明
导入 JSON读取本地的 liyaMessages.json，作为当前编辑基准。
编辑消息在浏览器内存中修改 id、条件、优先级、tags、lines 等字段。
校验检查重复 id、空 lines、未知 condition、结构问题。
命中预览使用固定 smoke test context，快速看不同情境会命中哪条消息。
触发反查反向分析每条消息在哪些模拟情境下可能被触发，是否被其他消息覆盖。
导出 JSON下载当前草稿为 liyaMessages.json，再手动替换项目里的正式文件。

2. 正确打开方式

不要直接双击 message-editor.html。如果地址栏是 file:///...，浏览器会因为 CORS 限制拦截 ES module，页面会停在“正在加载”。

推荐方式 A：VS Code Live Server

  1. 用 VS Code 打开项目根目录：D:/Project/birdwatch-text-sim
  2. 右键 message-editor.html
  3. 选择 Open with Live Server
  4. 浏览器地址应类似：http://127.0.0.1:5500/message-editor.html

方式 B：Python 本地服务器

cd D:/Project/birdwatch-text-sim
python -m http.server 8000

然后打开：

http://localhost:8000/message-editor.html

如果 Windows 里 Python 命令是 py：

py -m http.server 8000

3. 页面区域说明

顶部状态区

显示当前草稿数量、是否 fallback、校验错误、未覆盖数量、实际可触发数量等。

重点看：validation errors、unknown conditions、actual triggerable、covered only。

筛选区

按 id、lines、tags、conditions 搜索，也可以按 type、trigger、stage、condition、触发状态筛选。

命中预览

这是少量固定 dev context 的 smoke test。它用于快速确认“典型场景会命中哪条消息”。

消息列表

展示所有消息卡片。每张卡片能看到条件、优先级、文本 lines、触发状态和编辑按钮。

报告区

展示完整纯文本报告，包括基础校验、命中结果和触发反查总结。

导入 / 导出工具

导入本地 JSON 作为编辑基准；导出当前草稿为新的 liyaMessages.json。

4. 推荐工作流

1
打开编辑器
通过 Live Server 或 Python server 打开，确认页面不是 file://。
2
检查状态摘要
确认 loaded 为 true，usingFallback 为 false，draft messages 数量正确。
3
筛选或搜索消息
按 tags、conditions、触发状态快速找到要修改的消息。
4
点击“编辑”
修改 lines、conditions、priority、tags 等字段。
5
应用到草稿
编辑只存在当前页面内存里，不会自动保存到项目文件。
6
查看触发情况
确认消息是“可实际触发”，而不是“被覆盖”或“当前矩阵未匹配”。
7
导出 JSON
下载新的 liyaMessages.json，手动替换项目中的 data/liyaMessages.json。
8
进入游戏验证
重新打开游戏，确认发给妹妹后的回复符合预期。

5. 编辑字段说明

字段用途注意
id消息唯一标识。必须唯一，不要改成空。已有 queue item 可能保存了旧 messageId，正式数据中删除旧 id 前要谨慎。
type消息类型，例如 photo_reply。当前主要使用 photo_reply。
speaker说话人。力娅消息一般是 liya。
stage剧情阶段。当前主要是 early。
trigger.event触发事件。当前照片回复使用 photo_sent。
conditions触发条件。必须是 JSON 对象，例如 {"quality":"blurred"}。
priority优先级。越高越先被选中。特殊条件通常高于通用文本。
cooldown冷却字段。当前运行时还不消费它，更多是未来预留。
allowRepeat是否允许重复。当前 photo_reply 仍主要由选择器和 queue 固定 messageId；关系推进类建议谨慎。
delay延迟字段。当前正式流程仍保持旧 30 秒逻辑，这个字段暂时不是主控制源。
tags策划标签。用于筛选和整理，如“首次”“清晰”“小老师”“关系推进”。
lines实际聊天文本。每行一条，建议 1～3 行，不要过长。

6. conditions 可用字段

当前选择器只支持以下 condition key。新增其他 key 会被标记为 unknown condition，通常不会正常触发。

condition可用值示例
firstTimeSpeciestrue / false{"firstTimeSpecies":true}
repeatSpeciestrue / false{"repeatSpecies":true}
speciesId稳定鸟种 id 或数组{"speciesId":"light_vented_bulbul"}
timeOfDaymorning / day / dusk / night / unknown{"timeOfDay":"dusk"}
qualityclear / normal / blurred / unknown{"quality":"clear"}
compositioncentered / off_center / unknown{"composition":"off_center"}
不要把 mood、relationshipLevel、weather、schoolStatus 等当前选择器不支持的字段写进 conditions。可以先放到 tags，后续机制支持后再迁移。

7. 如何理解“触发情况总览”

触发情况总览回答的是：这条消息在模拟条件矩阵中有没有机会被玩家看到。

状态含义处理建议
可实际触发至少在某些模拟 context 中会被最终选中。正常。
条件匹配但被覆盖conditions 成立，但总是被更高 priority 或更高 specificity 的消息盖掉。检查 priority 是否过低，或条件是否太宽。
当前矩阵未匹配在当前 81 个模拟 context 中没有条件匹配。可能是条件太特殊，也可能是写错了条件值。
unknown conditionconditions 中存在选择器不支持的 key。改成支持字段，或先移到 tags。
“条件匹配”不等于“实际触发”。实际触发还要经过 priority、specificity 和稳定伪随机选择。

8. priority 建议

类型建议 priority说明
通用回复5～15没有特殊条件，用于兜底。
单一特殊条件35～60例如首次、重复、清晨、黄昏、模糊。
组合条件60～80例如首次 + 清晰 + 居中。
关系推进45～70不要过多盖掉普通回复，避免太频繁。

如果某条消息显示“条件匹配但被覆盖”，最常见原因是 priority 太低，或者同条件下有更具体的消息。

9. 导入、复制、导出的区别

操作作用是否改变项目文件
导入 JSON读取本地 JSON，替换当前页面内存草稿和当前基准。否
复制当前 JSON把当前草稿复制到剪贴板。否
导出 JSON下载当前草稿为 liyaMessages.json 文件。否，需要你手动替换
重置为当前基准 JSON丢弃页面内草稿修改，恢复到最近一次导入或初始加载的数据。否

10. 常见问题

页面一直显示“正在加载…”

优先检查地址栏。如果是 file:///...，说明你直接双击打开了 HTML，需要改用 Live Server 或 Python server。

导入后出现 unknown condition

说明某条消息的 conditions 使用了当前选择器不支持的 key。先改成支持字段，或移到 tags。

某条消息“条件匹配但被覆盖”

说明它的条件成立，但最终被更高 priority 或更具体的消息选走。可以调整 priority，或者把 conditions 写得更具体。

导出了 JSON，游戏里没有变化

导出只是下载文件。你还需要手动用导出的文件替换项目中的 data/liyaMessages.json，然后刷新游戏页面。

能不能直接在编辑器里保存到项目文件？

当前不支持。浏览器普通页面不能直接写回项目文件，这也是为了避免误覆盖正式数据。

11. 推荐检查清单

  • validation errors 为 0。
  • unknown conditions 为 0。
  • 重要消息不是“当前矩阵未匹配”。
  • 关系推进消息不要过多“可实际触发”，避免太频繁。
  • 通用消息 priority 不要太高，避免盖掉特殊回复。
  • 导出前确认 dirty 状态和报告区提示。
  • 导出后手动替换 data/liyaMessages.json，再进入游戏验证。
