# 《认鸟手信》当前 UI / CSS / 交互流程现状

更新时间：2026-06-05

本文档用于给交互设计师 / UI 设计师 agent 快速理解当前界面现状。它不是 DevLog，不是代码地图，也不是需求文档。如果本文档与实际代码冲突，以实际代码为准。

## 1. 当前界面总览

- 当前游戏是偏手机端、小屏优先的原生 HTML / CSS / JavaScript 单页网页游戏。
- `index.html` 提供静态骨架，`src/main.js` 负责运行时编排、状态切换和大部分界面渲染。
- `styles/style.css` 承担主游戏全部样式，包括状态卡、消息面板、笔记页、FOCUS 拍摄界面、拍立得、结算和动画。
- `src/ui/messagePanel.js` 负责消息面板 UI 片段与聊天滚动 / 分句动画辅助。
- `src/ui/fieldGuidePanel.js` 负责笔记 / 图鉴面板 HTML 片段。
- `message-editor.html` 是离线消息编辑工具，不是主游戏页面，也不接入 `index.html` 的运行时流程。

## 2. `index.html` 静态骨架

当前静态骨架主要由以下区域组成：

1. `#app-loading`
   - 启动时的 loading mask。
2. `.page`
   - 整个主页面容器。
3. `.site-header`
   - `#gameTitle` 与副标题。
4. `.status-grid`
   - 顶部状态区。
   - 其中 `#modeText`、`#spotText` 实际是按钮卡片骨架，文案和状态由运行时改写。
5. `.event-box`
   - `#eventText` 事件描述主阅读区。
6. `#actionPanel`
   - 行动按钮区，完全由 `main.js` 动态填充。
7. `.layout`
   - 左侧 / 上方 `#detailPanel`。
   - 右侧 / 下方 `.log-panel` 与 `#logList`。
8. `<script type="module" src="./src/main.js">`
   - 唯一正式启动入口。

当前还维护了一组 `modulepreload`，预取 `src/main.js`、`src/core/*`、`src/gameSession.js`、`src/gameState.js`、`data/config.js` 等模块。

静态存在的只是骨架。消息入口区 `.utility-actions`、底部系统区 `.reset-actions`、消息面板内容、笔记面板内容、结算详情、拍摄 playfield 等主要内容都由 `main.js` 在运行时插入。

## 3. 主界面布局层级

玩家常见的阅读顺序大致是：

1. 标题区
   - 当前显示“第几天”的观察标题。
2. 顶部状态区 `.status-grid`
   - 模式 / 时间与电量 / 消息入口 / 方向 / 位置 / 拍摄时机。
3. 事件描述栏 `.event-box`
   - 当前叙事文本主阅读区。
4. 运行时工具入口 `.utility-actions`
   - 位于事件描述下、行动按钮上。
   - 当前主要是“查看消息 / 打开笔记”。
5. 行动按钮区 `#actionPanel`
   - 跟随 mode / photoPhase 切换。
6. 详情区 `#detailPanel`
   - 默认可为空。
   - 打开消息或笔记时成为 inline panel。
   - 结算、拍摄、初见等阶段也会占用这里。
7. 观察日志 `.log-panel`
   - 持续显示观察日志。
8. 底部系统操作 `.reset-actions`
   - 运行时创建。
   - 与笔记面板内部的“清除记录 / 确认清除”是不同层。

当前不是多页面路由，而是 `main.js` 依据 `gameState.mode`、`photoPhase`、`activeOverlay` 做单页状态切换。消息与笔记是互斥 inline panel，不会同时展开。

## 4. 当前视觉语言 / CSS 风格

当前视觉语言可以概括为：

- 纸感、手账、自然观察记录、轻复古。
- 背景不是纯平色，使用暖色渐变、纸纹理、投影和轻微滤镜。
- 主色偏纸黄、棕褐、苔绿、暗金，强调“笔记本 / 观察卡 / 拍立得”质感。
- 重要 UI 都有明显卡片层级：
  - 顶部状态卡。
  - 事件纸条。
  - 消息纸页。
  - 笔记文件夹。
  - 拍立得照片。
  - 结算摘要卡。
- `:root` 中已定义较多颜色 token，包括纸张、墨色、强调色、消息未读色、徽章色等。
- 主按钮、工具入口按钮、状态卡按钮是同一套卡片式按钮语言，视觉上不是强侵入的现代 app CTA，更像纸面操作件。
- `new` / 未读使用更暖、更亮的提示色，语义上是“有新内容可看”，不是数值奖励。
- `@media (prefers-reduced-motion: reduce)` 已覆盖主要 reveal、拍立得、FOCUS、消息分句等动效，说明当前实现已经把 reduced motion 当成正式边界，而不是附带优化。

## 5. 核心 UI 区域说明

### 5.1 顶部状态区

- 由 `.status-grid` 组成。
- 当前显示内容包括：
  - 模式或笔记入口按钮。
  - 当前时间与当前电量的组合卡。
  - 消息入口按钮。
  - 当前方向。
  - 当前位置。
  - 拍摄时机。
- 这些内容多数不是玩家直接编辑的状态，而是运行时派生显示。
- `#photoTimingText` 会在 `PHOTO` 流程中跟随 `DECISION / FOCUS / RESULT / REPOSITION / LOST` 变化。
- FOCUS 激活时顶部“拍摄时机”卡会进入更强提示态，说明顶部状态区兼具信息和节奏提示功能。

### 5.2 事件描述栏

- `#eventText` 是当前最核心的阅读节点。
- `.event-box` 左侧有明显竖向强调色带。
- 普通事件、开场独白、`FIRST_ENCOUNTER`、结算文案使用不同 reveal 路径。
- 当前实现里：
  - 开场独白按段 reveal。
  - `FIRST_ENCOUNTER` 文本按中文句号切段，逐段淡入上移。
  - 普通事件文本支持强调词与常规 reveal。
- 事件文本 reveal 不是每次 render 都应该重播，当前运行时通过 reveal key 控制，避免动画 cleanup 或二次 render 造成文本跳动。

### 5.3 行动按钮区

- `#actionPanel` 完全由当前 `mode / photoPhase` 决定。
- 常见语义包括：
  - 开始游戏。
  - 选择起始鸟点。
  - 转向、等待、前往、远听。
  - 举起相机。
  - 按下快门。
  - 继续跟焦、再等一等、放弃。
  - 提前撤离并结算。
- 当前按钮权重有明显区分：
  - 主动作维持更强视觉优先级。
  - 放弃 / 退出 / 提前结算属于弱化动作，不应被随意提升为默认主按钮。

### 5.4 消息入口与消息面板

- 消息入口按钮在运行时 `.utility-actions` 中创建。
- 打开后，`#detailPanel` 渲染为消息面板，由 `src/ui/messagePanel.js` 负责主要 HTML 片段。
- 当前消息面板包含两层：
  - 线程列表。
  - 线程聊天页。
- 当前线程至少包含 Liya，也保留母亲、喵喵等初始线程入口。
- 聊天页支持：
  - 玩家发送的照片消息。
  - Liya 回复气泡。
  - 多行分句播放。
  - 未读分隔线 `chat-unread-divider`。
  - 引用拍立得卡片。
  - 滚动位置锚点恢复。
- 当前已读边界非常重要：
  - 打开消息列表不等于已读。
  - 进入 Liya 聊天也不自动等于已读全部。
  - 只有“到期的 Liya 回复在聊天视口内真实可见”才会清未读并写回状态。
- `messagePanel.js` 当前只负责 UI 与动画辅助，不负责业务存档或消息选择逻辑。

### 5.5 笔记 / Field Guide 面板

- 玩家可见名称是“笔记”，内部代码大量仍叫 `fieldGuide`。
- 打开后同样占用 `#detailPanel`，与消息面板互斥。
- `src/ui/fieldGuidePanel.js` 提供：
  - 空态页。
  - 鸟种列表页。
  - 卡牌详情页。
  - 清除记录确认页。
- 笔记当前包含：
  - 已发现鸟种列表。
  - 已收集卡牌。
  - 卡牌详情页。
  - 同卡多张 snapshot 翻页。
  - 发送给妹妹。
  - 妹妹补充知识。
  - 底部“关闭手册”按钮。
- 知识状态至少区分 `UNKNOWN / HEARD / SEEN / CATALOGUED`。
- 未加新前不应泄露正式鸟名。
- `new` 标记来自 field guide 实际新卡状态，不是单纯入口活跃态。
- `fieldGuidePanel.js` 只拼 UI，不负责业务写入。

### 5.6 `PHOTO / FOCUS / RESULT` 拍摄界面

- 拍摄流程当前明确分为：
  - `DECISION`
  - `FOCUS`
  - `RESULT`
  - `REPOSITION`
  - `LOST`
- `FOCUS` 阶段包含：
  - `.focus-playfield`
  - `.focus-frame`
  - moving badge
  - 快门反馈闪光
- 按下快门后会出现拍立得 overlay，随后退场。
- `RESULT` 阶段允许继续跟焦、再等、放弃。
- `REPOSITION` 表示鸟还在视野里，但离开当前取景位置。
- `LOST` 表示这次已经失去它的位置。
- 当前所见即所得边界仍然成立：
  - 按下快门瞬间可见的行为态决定抽卡 rarity 权重。
  - 对焦结果来自当时可见位置与焦框判定。
  - scale / rotate 进入 snapshot 显示信息，但不参与对焦命中本身。

### 5.7 结算界面

- `SETTLEMENT` 不是一上来就平铺全部内容，而是带有摘要折叠 / 展开节奏。
- “今天的收获”有 reveal 过程。
- 新卡会在结算照片列表中保持显著的新内容提示。
- 当前界面还保留问卷入口和继续到下一天的动作位。
- 需要注意的是：
  - survey / analytics 的默认开关来自 `data/config.js`。
  - 默认 main 不应因为界面存在这些区域就自动显示问卷或自动发送 analytics。

### 5.8 观察日志与底部系统操作

- `.log-panel` 常驻在主布局中，显示观察日志。
- 用途是把玩家行动和观察结果串成可回看记录。
- 运行时还会创建底部系统操作区 `.reset-actions`。
- 清除记录确认当前是独立确认面板，由 `renderResetSaveConfirmPanel()` 产出，不应误并回普通笔记内容页。

## 6. 当前主要交互流程

### 6.1 开始游戏流程

1. 打开页面，先看到 loading。
2. `main.js` 初始化后隐藏 loading，进入 `START`。
3. 若 opening survey / tester profile 开关开启，会先出现对应开场门槛。
4. 进入起始鸟点选择 `START_SPOT_SELECT`。
5. 选定起始鸟点后进入正式探索。

### 6.2 探索流程

1. 玩家查看当前鸟点与方向。
2. 可执行转向、等待、远听、前往其他鸟点等动作。
3. 系统根据当前 spot、方向、时间和遇鸟规则推进事件。
4. 发现鸟后进入初见或拍摄流程。

### 6.3 初次遇见与加新流程

1. 首次近距离遇到未知鸟时，进入 `FIRST_ENCOUNTER`。
2. 该阶段先给匿名化描述，不提前泄露正式鸟名。
3. 继续后进入 `PHOTO`。
4. 拍到卡牌后，可在笔记详情把照片发给妹妹。
5. 妹妹回复被阅读并满足条件后，才会推动妹妹补充知识与自动加新。
6. `CATALOGUED` 后，正式鸟名才稳定进入图鉴层面的可见状态。

### 6.4 拍摄流程

1. `PHOTO / DECISION` 决定举机、等待或放弃。
2. 进入 `FOCUS` 后，玩家在 moving badge 与焦框节奏中寻找时机。
3. 按下快门时捕获可见行为态、对焦结果和拍立得视觉参数。
4. 进入 `RESULT`，可继续跟焦、再等或放弃。
5. 鸟偏移则进 `REPOSITION`，彻底丢失则进 `LOST`。
6. 电量或回合条件到达终点后进入 `SETTLEMENT`。

### 6.5 消息流程

1. 在卡牌详情中发送给妹妹。
2. 系统生成对应 queue item，并进入 1-2 秒当前运行时延迟。
3. 到期后消息入口出现未读提示。
4. 玩家打开消息列表，再进入 Liya 线程。
5. 回复按多行逐句播放。
6. 只有到期回复实际可见时才视为已读。
7. 已读转换后，才可能解锁妹妹补充与自动加新后续。

### 6.6 笔记流程

1. 玩家从工具入口打开笔记。
2. 在列表中浏览已发现鸟种。
3. 进入某张卡牌详情。
4. 查看拍立得、翻看同卡多张 snapshot。
5. 需要时发送给妹妹。
6. 若已解锁，可查看妹妹补充。
7. 通过底部按钮关闭笔记。

### 6.7 Liya 消息编辑器流程

- `message-editor.html` 是独立离线工具，不是游戏内 panel。
- 当前页面支持：
  - 加载默认数据。
  - 导入 JSON。
  - 粘贴导入。
  - 搜索与快速筛选。
  - 实时编辑。
  - 当前消息校验与全局校验。
  - 预览。
  - 新增、复制、删除。
  - 导出 JSON。
- 它默认尝试读取 `data/liyaMessages.json`，失败时提示手动导入。
- 编辑发生在浏览器内存中，导出只是下载 JSON，不会自动写回项目文件，也不影响主游戏运行时。

## 7. 关键状态与 mode / phase 对照

| mode / phase | 玩家看到的主要界面 | 主要行动 | 交互注意 |
|---|---|---|---|
| `START` | 开场说明、标题、事件文案 | 开始、继续开场 | 可能叠加 opening survey / tester profile 门槛 |
| `START_SPOT_SELECT` | 起始鸟点选择说明 | 选择起点 | 属于正式探索前的起步配置 |
| `EXPLORE` | 常规状态卡、事件栏、行动按钮、日志 | 转向、等待、前往、远听 | 主循环模式 |
| `DISTANT_LISTEN` | 倾听 / 远听相关说明 | 选择继续听、返回等 | 仍是探索子流程，不是独立页面 |
| `FIRST_ENCOUNTER` | 强叙事事件描述 | 继续 | 文本分段 reveal，且不泄露正式鸟名 |
| `PHOTO / DECISION` | 拍摄准备态、时机提示 | 举机、等待、放弃 | 还没进入 FOCUS 动态 playfield |
| `PHOTO / FOCUS` | 焦框、moving badge、快门反馈 | 按快门、超时 | 所见即所得的关键阶段 |
| `PHOTO / RESULT` | 拍后结果说明、拍立得反馈 | 跟焦、再等、放弃 | 允许继续追拍 |

| `SETTLEMENT` | 收获摘要、照片列表、问卷入口、继续动作 | 展开收获、继续、问卷相关动作 | 默认开关下不应自动显示 survey / 发送 analytics |
| `FIELD_GUIDE` | 笔记页 | 浏览、加新、查看详情 | 当前玩家可见入口主要通过 `activeOverlay === "fieldGuide"` 打开，代码里 `FIELD_GUIDE` mode 仍存在 |

## 8. 当前不可误改的 UX 边界

- 未加新前不泄露正式鸟名。
- 玩家可见名是“笔记”，不要为了界面统一就把所有玩家文案改成 `fieldGuide`。
- 打开消息列表不等于已读。
- 妹妹补充不是发出去立刻显示，仍受回复阅读与后续流程控制。
- 自动加新不能在消息刚到或刚打开列表时就强行完成。
- `PHOTO / FOCUS / RESULT` 的节奏不要随意合并成一个模糊拍照态。
- 当前 `FOCUS` 阶段没有“再等一等”主按钮，不要凭习惯补回。
- 焦框视觉位置必须和实际判定一致。
- scale / rotate 进入 snapshot 表现，但不参与对焦命中规则。
- 放弃 / 退出 / 提前结算不应提升为默认主按钮。
- `message-editor.html` 不接入主游戏入口。
- survey / analytics 默认关闭，不要因为做 UI 调整误触发。
- `saveManager-lite` 和 `telemetryAdapter-lite` 只是轻封装，不代表这些系统已完全模块化完毕。

## 9. 当前可能需要交互设计师介入的问题

- `main.js` 仍是页面级编排中心，界面状态很多，后续大改 UI 时需要先判断改动是不是会碰到业务边界。
- 小屏下“顶部状态区 + 事件描述 + 工具入口 + 行动按钮 + inline panel + 日志”的信息密度较高，可能需要重新评估层级压力。
- FOCUS 的进入、失败、继续跟焦节奏是否足够直观，仍可能需要专门讨论。
- 消息未读、分句播放、引用拍立得、自动加新之间的阅读仪式感较强，但理解成本也偏高，可能需要设计师参与整理信息顺序。
- 笔记卡牌详情的信息密度较大，拍立得、统计、发送按钮、妹妹补充之间的优先级还有讨论空间。
- 结算页同时承载收获、问卷入口、继续动作，层级是否足够清楚，可能需要进一步讨论。
- 三测如果加入天气、地图、鸟点解锁，顶部状态区很可能需要重新设计承载方式。
- 未来发布系统若引入，必须防止它吞掉当前“妹妹 / 观鸟 / 叙事”为核心的体验重心。

## 10. 给交互设计师 agent 的协作建议

后续讨论建议优先关注：

1. 当前操作是否符合玩家预期，尤其是消息已读、发送给妹妹、自动加新之间的因果链。
2. 顶部状态、事件文本、工具入口、行动按钮的层级是否清楚。
3. 主按钮 / 次按钮 / 退出按钮的权重是否准确。
4. 消息与笔记的入口是否足够容易理解。
5. FOCUS 失败、继续跟焦、失去目标是否容易读懂。
6. 自动加新是否有足够仪式感，同时不过度打断。
7. 小屏阅读与点击负担是否舒适。
8. 三测增加天气 / 地图 / 发布系统时，如何扩容而不破坏当前简洁性。

同时要避免：

- 直接提出大范围视觉重构。
- 把游戏改成社交媒体经营模拟器。
- 把 `message-editor.html` 与主游戏 UI 混为一体。
