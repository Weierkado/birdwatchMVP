2. 事件系统概览

事件系统在探索阶段被动提示玩家非当前朝向的鸟活动——让玩家在决定转向前就能感知到「左侧有点什么」。通知用独立元素短暂显示，与主叙事文本分离，不替换 #eventText。

核心职责

· 探索方向提示：非朝向方向的鸟活动感知

· 天气变化通知：每局一次切换时推送

· 未来扩展：特殊地点发现、限时遭遇前摇

核心约束

· 只在 mode === 'EXPLORE' 时触发

· 单槽位显示，先进先出队列，容量 3 条

· 出现时闪烁两次，稳定显示后自动消失

· 不替换 #eventText，与主叙事文本互不干扰

3. 方向提示原理

所有鸟实例都有 bird.directionIndex（0-3），与玩家的 facingDirection 使用同一坐标系。非朝向方向的鸟即为「侧面」或「背后」的线索来源。

// 相对方向计算 const left = (facingDirection + 3) % 4 // 左侧 const right = (facingDirection + 1) % 4 // 右侧 const behind = (facingDirection + 2) % 4 // 身后（当前不生成提示，保留扩展） // 取侧面活跃鸟 getBirdsAtDirection(state, dirIndex) → state.activeBirds.filter(b => b.directionIndex === dirIndex)
事件提示的方向感知链路： state.activeBirds └── 按 directionIndex 分组 ├── facingDirection → 已由 generateClues() 处理，不重复生成事件 ├── left direction → 左侧鸟 → 生成「左侧」提示文本 ├── right direction → 右侧鸟 → 生成「右侧」提示文本 └── behind direction → 背后鸟 → 暂不生成（避免引导玩家原地转两圈）

方向名称已由 getSurroundingSpotMap(state) 提供（left / right 字段），可直接取用。

为什么不提示「背后方向」：背后 = 玩家已经看过或刚刚离开的方向，提示引导价值低，且容易驱动玩家原地来回转向刷提示。后续如果有「忽然背后有声音」的特殊遭遇，再单独设计。

hintType 字段（新增到 data/species.js）

在每个物种的定义对象里新增可选字段 hintType，默认为 'sound'：

// data/species.js — 示例 { id: 'kingfisher', hintType: 'shadow' }, // 翠鸟：转瞬即逝，只见鸟影 { id: 'sparrow', hintType: 'sound' }, // 麻雀：叽叽喳喳，声音明显 { id: 'red_billed_magpie', hintType: 'sound' }, // 红嘴蓝鹊：鸣声响亮 { id: 'mandarin_duck', hintType: 'movement' }, // 鸳鸯：水面动静 { id: 'blackbird', hintType: 'sound' }, // 乌鸫：鸣声好辨别 { id: 'night_heron', hintType: 'shadow' }, // 夜鹭：静默，只留轮廓
hintType 反映的是鸟的气质，不是难度参数——翠鸟和夜鹭都用 shadow，但原因不同：翠鸟是因为它快，夜鹭是因为它静。这让文本感觉更像真实观鸟而不是游戏数值。

5. 触发规则

前置条件

  • 只在 state.mode === 'EXPLORE' 时触发——进入 PHOTO / DISTANT_LISTEN / SETTLEMENT 等模式时不触发
  • 天气变化事件（WEATHER_CHANGE）不受 mode 限制，任何模式都可推送（但仍走事件队列）

触发时机

玩家行动来源函数是否扫描侧面鸟触发概率
转向（turnLeft / turnRight）turnDirection()是70%（转向后刚好侧面有鸟，高概率感知）
抵达新鸟点handleDistantListenAction()是85%（新环境，感知最敏锐）
等待（wait）handleExploreAction → 'wait'是30%（静候时偶尔感知侧面）
observe（观察当前方向）handleExploreAction → 'observe'否—（专注正前方，不感知侧面）
listenDistanthandleExploreAction → 'listenDistant'否—（专注远处，已有另一套线索系统）

扫描逻辑

// eventSystem.js — scanSideEvents(state) scanSideEvents(state) { if (state.mode !== 'EXPLORE') return const leftDir = (state.facingDirection + 3) % 4 const rightDir = (state.facingDirection + 1) % 4 for (const [dir, label] of [[leftDir, 'left'], [rightDir, 'right']]) { const birds = state.activeBirds.filter(b => b.directionIndex === dir) if (birds.length === 0) continue if (!this.canTrigger(state.currentSpotId, dir)) continue const species = getSpeciesById(birds[0].speciesId) const hintType = species?.hintType || 'sound' const dirText = label === 'left' ? '左侧' : '右侧' const text = HINT_TEXTS[hintType][Math.floor(Math.random() * HINT_TEXTS[hintType].length)] .replace('{dir}', dirText) this.dispatch({ type: hintType.toUpperCase(), text, spotId: state.currentSpotId, dirIndex: dir }) break // 每次扫描最多生成一条，避免同时「左侧右侧」双提示 } }

冷却

同一鸟点、同一方向，在 EVENT_HINT_COOLDOWN_TURNS（占位值：3 回合）内不重复生成提示。以回合数计而非毫秒，符合回合制游戏节奏。

// 冷却记录（Map，每回合回滚检查） cooldowns: Map<`${spotId}:${dirIndex}`, lastTriggerTurn> canTrigger(spotId, dirIndex) { const last = this.cooldowns.get(`${spotId}:${dirIndex}`) ?? -99 return (state.currentTurn - last) > EVENT_HINT_COOLDOWN_TURNS }

8. 天气系统概览

天气是每局级别的全局状态（一局 = 一天 = 30 回合）。在局开始时初始化，中段有一次切换机会。天气影响四个维度：

体力消耗 → 回合消耗

MVP 没有独立体力条——「体力消耗」对应移动鸟点的 travelCost。天气加乘在移动时额外扣回合。原地行动（观察、转向）不受影响。

鸟类活动

修改 PHOTO_SEQUENCE_CONFIG_BY_SPECIES 中各物种的行为状态权重（NORMAL/INTERESTING/REMARKABLE）和飞走概率，体现鸟在不同天气下的活跃程度。

拍摄背景

照片 snapshot 记录 weatherKey 字段，用于拍立得视觉渲染时加载对应背景。框架阶段可以先用 CSS class 名占位。

拍摄滤镜

snapshot 同时记录 weatherFilterConfig，照片显影时叠加视觉滤镜。P2 优先级，等美术资产后实装；框架阶段写入字段但不渲染。

天气信息的 UI 显示

现有 #spotText 是一个 aria-disabled 的占位按钮（「查看消息」），目前没有实际功能。天气信息借用这个槽位：将 #spotText 改为 span 标签（去掉按钮语义），显示当前天气，如「☀ 晴天」。

<!-- index.html: 将原 button 改为 span --> <div class="status-item status-spot"> <span class="status-label">天气</span> <span class="status-value" id="spotText">☀ 晴天</span> </div>
与 renderStatusBlocks() 的关系说明：当前 MVP renderStatusBlocks()（main.js 约 L3820）已经向 elements.spot（即 #spotText）写入「天气 / 晴天」文本——这是游戏初始化时的静态标签写入。WE-C01（天气 UI 初始化）不需要新增额外的 DOM 渲染调用，只需：① 修改 index.html 将 button → span；② weatherSystem.initForSession() 调用后由 renderStatusBlocks() 自然渲染初始天气（确保 elements.spot.textContent 读取 weatherSystem.currentLabel 而非硬编码「晴天」）；③ 天气切换时 weatherSystem 直接 document.getElementById('spotText').textContent = label 更新（§10 已有此代码）。不要在 renderStatusBlocks() 之外再加重复渲染逻辑。
这是对「查看消息」按钮的破坏性修改。若「查看消息」功能（妹妹消息面板入口）后续要放在这里，需要提前协调放置位置。当前版本该按钮无实际功能，改动可以直接进行。

9. 四种天气定义

CLEAR · 晴天

☀ 晴天

移动额外回合消耗：+0

鸟活动调整：基准（无修改）

拍摄背景 key：weather_clear

初始概率权重（占位）：40

CLOUDY · 阴天

☁ 阴天

移动额外回合消耗：+0

鸟活动调整：整体 INTERESTING/REMARKABLE 权重 -10%

拍摄背景 key：weather_cloudy

初始概率权重（占位）：30

RAIN · 雨天

🌧 雨天

移动额外回合消耗：+1（消耗更多体力）

鸟活动调整：陆鸟飞走概率 +0.04；水鸟（鸳鸯）驻留时间 +2 回合

拍摄背景 key：weather_rain

初始概率权重（占位）：20

FOG · 雾天

🌫 雾天

移动额外回合消耗：+0

鸟活动调整：clueStrength 初始值 -1（鸟线索更难感知）

拍摄背景 key：weather_fog

初始概率权重（占位）：10

雨天额外 +1 移动回合消耗让玩家在下雨时更倾向于原地等待而非频繁换鸟点，自然契合「下雨天不太想走动」的体验。雾天线索削弱（clueStrength 初始 -1）让 observe 时更难找到鸟，传达「大雾能见度低」而不是用一个独立的 FOG_HIDE_RADIUS 来处理。

10. 初始化与切换

初始化

weatherSystem.initForSession() 在 startGameAtSpot() 最后调用，加权随机确定当天初始天气，并更新 #spotText。

// src/weatherSystem.js initForSession() { this.current = weightedRandom(WEATHER_WEIGHTS) this.switched = false document.getElementById('spotText').textContent = WEATHER_PROFILES[this.current].label } const WEATHER_WEIGHTS = { CLEAR: 40, CLOUDY: 30, RAIN: 20, FOG: 10 } // weightedRandom 对权重归一化后抽取，不需要手动保证和为 100

切换

在每次 advanceTurn(state) 之后调用 weatherSystem.trySwitch(state.currentTurn)，检查是否触发切换：

trySwitch(currentTurn) { if (this.switched) return if (currentTurn < WEATHER_SWITCH_TURN_MIN || currentTurn > WEATHER_SWITCH_TURN_MAX) return if (Math.random() > WEATHER_SWITCH_PROB_PER_TURN) return const next = pickNewWeather(this.current) // 不允许切换到同种 const text = CHANGE_TEXTS[this.current]?.[next] || '【天气有了变化】' this.current = next this.switched = true document.getElementById('spotText').textContent = WEATHER_PROFILES[next].label eventSystem.dispatch({ type: 'WEATHER_CHANGE', text, priority: 'high' }) } // 常量（data/config.js） WEATHER_SWITCH_TURN_MIN: 9, // 切换窗口开始（第 9 回合，约 30%） WEATHER_SWITCH_TURN_MAX: 21, // 切换窗口结束（第 21 回合，约 70%） WEATHER_SWITCH_PROB_PER_TURN: 0.07, // 每回合触发概率，约 50% 局数会发生切换
